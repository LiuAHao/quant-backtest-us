# 策略回测入口、任务日志与批量删除改造方案

## 1. 背景

当前系统已经具备策略管理、异步回测、事件分析与报告中心能力，但在实际使用中存在三个明显体验缺口：

1. 策略管理页的“运行回测”会直接按默认参数启动任务，跳过了用户确认参数的步骤。
2. 回测报告与事件分析结果没有展示运行过程中的任务日志，失败时很难快速定位问题。
3. 策略/事件定义管理仍然以单条删除为主，缺少高效的批量清理能力。

本次改造目标是在不破坏现有代码分层的前提下，沿用现有 `frontend/src/App.jsx -> frontend/src/api.js -> backend/api/* -> backend/services/*` 结构补齐这些能力。

## 2. 问题拆解

### 2.1 策略管理“一键运行回测”行为不符合预期

现状：

- 前端 `runStrategy(strategy)` 直接调用 `createBacktest(strategy, backtestDefaults)`。
- 默认参数来自系统设置和最近交易日，因此点击后会立刻创建一个“近一个月”的异步任务。

问题：

- 用户失去了对日期、资金、手续费、滑点的确认权。
- 容易产生误跑任务和无效报告。

改造思路：

- 保留策略管理页上的“运行回测”入口，但不再直接创建任务。
- 改为跳转到“新建回测”页，并把当前策略 ID 作为预选值注入表单。
- 新建回测页仍然保留模板、日期区间和资金参数编辑能力，最终只有用户点击“开始异步回测”才真正建任务。

### 2.2 报告与结果页缺少运行日志

现状：

- 回测引擎与部分策略内部已经使用 `loguru` 输出日志。
- 服务层与事件分析层也有部分 `logging` / `logger.exception` 输出。
- 但这些日志只写到控制台或文件，没有进入任务结果结构，也没有在前端详情页展示。

问题：

- 成功任务无法复盘关键阶段。
- 失败任务虽然有 `error_message`，但缺少上下文。
- 用户只能猜测问题出在策略逻辑、数据读取、撮合执行还是事件扫描。

改造思路：

- 在服务层引入“任务级日志采集器”，同时捕获：
  - 当前 worker 线程内的 `loguru` 日志
  - 当前 worker 线程内的标准库 `logging` 日志
- 每个回测任务、事件分析任务执行期间都挂载该采集器。
- 任务结束后把日志：
  - 写入任务表中的 `runtime_logs_json`
  - 回填到回测 JSON 报告 / 事件分析 JSON 结果的 `runtime.logs`
- 前端详情页新增“运行日志”板块，放在结果展示的末尾，按时间顺序显示。
- 对失败/取消任务放开详情查看能力，只要任务不在 `queued/running`，都可以打开详情页看错误信息和日志。

### 2.3 目前日志设计是否完善

结论：目前只能算“基础可用”，还不算完善。

已有优点：

- 策略层已经普遍使用 `loguru`，输出门槛低。
- 服务层已有异常日志，失败能留下痕迹。

主要不足：

1. 日志缺少统一任务上下文。
   - 原先没有按任务维度持久化，同一时间多个任务时不利于隔离分析。
2. 日志结构还不够语义化。
   - 大多数仍是文本日志，没有标准 `stage`、`component`、`ts_code`、`trade_date` 等字段。
3. 成功和失败场景落盘不一致。
   - 原先只有成功任务有报告文件，失败任务只能看一句错误。
4. 报告侧没有最终诊断入口。
   - 用户看完指标后无法直接回看执行过程。

本次改造解决的是“任务级采集 + 报告级展示”。

后续建议的增强方向：

1. 为关键阶段补结构化埋点。
   - 如 `prepare_data`、`strategy_init`、`match_orders`、`generate_report`、`event_scan`、`attach_returns`。
2. 为异常增加统一错误码和上下文字段。
3. 在策略模板与事件模板文档中约定推荐日志写法。
4. 对超长日志做分级和截断策略，避免前端加载过重。

## 3. 方案设计

### 3.1 前端改造

涉及文件：

- `frontend/src/App.jsx`
- `frontend/src/api.js`
- `frontend/src/styles.css`

改造点：

1. 新增“预选策略”状态。
   - 从策略管理点击“运行回测”时，仅设置 `preferredBacktestStrategyId` 并跳转到 `new_backtest`。
   - 新建回测页通过 `useEffect` 自动选中对应策略。
2. 调整回测/事件分析详情页的加载条件。
   - 不再只允许 `success` 打开。
   - `failed`、`cancelled` 也可以进入详情查看日志。
3. 在回测报告页与事件分析结果页追加“运行日志”面板。
4. 在策略管理页与事件定义管理页新增“多选删除”模式。
   - 进入多选模式后显示勾选框。
   - 顶部展示“删除已选 N 项”操作。
   - 保持单条删除按钮逻辑不变，多选模式下仅切换为选择动作。

### 3.2 后端改造

涉及文件：

- `backend/services/task_logging.py`
- `backend/services/backtest_service.py`
- `backend/services/event_analysis_service.py`
- `backend/services/report_service.py`
- `backend/api/strategies.py`
- `backend/api/event_definitions.py`
- `backend/db/database.py`
- `backend/schemas.py`
- `backtest/engine.py`

改造点：

1. 数据库迁移
   - `backtest_tasks.runtime_logs_json`
   - `event_analysis_tasks.runtime_logs_json`

2. 任务级日志采集
   - 新增 `TaskLogCapture`
   - 按线程过滤日志，避免并发任务串日志
   - 支持追加异常堆栈

3. 回测任务链路
   - 在 `BacktestService._run_task()` 中包裹日志采集
   - 回测成功时写入任务表，并通过 `BacktestEngine.set_report_enricher()` 把日志注入报告 payload
   - 回测失败时也把日志写进 `runtime_logs_json`

4. 事件分析任务链路
   - 在 `EventAnalysisService._run_task()` 中包裹日志采集
   - 成功时把日志写入结果 JSON 的 `runtime.logs`
   - 失败时把日志写入任务表，供详情接口返回

5. 报告/结果接口增强
   - 回测报告接口即使没有 JSON 文件，也会返回任务级 runtime 信息
   - 事件分析结果接口同样保证失败任务可拿到日志

6. 批量删除接口
   - `POST /api/strategies/batch-delete`
   - `POST /api/event-definitions/batch-delete`
   - 返回 `deleted_ids` 与 `failed`，便于前端做部分成功提示

## 4. 风险与兼容性

### 4.1 兼容性

- 原有单条删除接口保留，不影响旧调用。
- 回测结果与事件分析结果原有 JSON 结构只新增 `runtime` 字段，属于向后兼容扩展。
- 新增数据库字段通过迁移函数补齐，不要求清库。

### 4.2 风险点

1. 并发任务日志串线风险
   - 通过线程 ID 过滤已做第一层隔离。
   - 若未来任务执行模型改成协程或子进程，需要重新设计采集上下文。
2. 日志过大
   - 当前采用内存队列上限控制，避免无限增长。
3. 前端详情页负载增加
   - 日志列表做滚动容器，不与图表区域混在一起。

## 5. 验证建议

### 5.1 回测入口

1. 在策略管理页点击任意策略“运行回测”
2. 应跳转到“新建回测”
3. 对应策略应默认高亮选中
4. 日期、资金、手续费、滑点仍可手动修改
5. 未点击“开始异步回测”前，不应创建新任务

### 5.2 回测日志

1. 运行一个正常回测
2. 打开详情页
3. 页面底部应能看到运行日志
4. 报告导出的 JSON 中应包含 `runtime.logs`

### 5.3 事件分析日志

1. 运行一个事件分析任务
2. 打开结果详情
3. 页面底部应能看到运行日志
4. 结果 JSON 中应包含 `runtime.logs`

### 5.4 失败诊断

1. 人为制造一个策略或事件分析运行异常
2. 任务结束后应允许打开详情
3. 页面应展示错误信息和对应日志

### 5.5 批量删除

1. 进入策略管理页多选模式
2. 勾选多个策略
3. 点击删除已选
4. 已删除项从列表移除
5. 如存在运行中任务导致部分删除失败，应显示部分成功提示

## 6. 后续建议

如果下一轮继续完善，我建议优先做两件事：

1. 把日志从“文本记录”进一步升级为“结构化阶段事件”。
2. 把批量删除能力扩展到回测结果、事件分析任务和报告中心，形成统一的列表操作体验。
