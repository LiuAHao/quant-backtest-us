# 页面导航、弹窗互斥与事件分析稳定性修复方案

> **状态：已完成** (2026-05-06)

## 1. 背景

当前系统已经具备策略管理、回测结果、报告中心、事件分析与任务中心能力，但在连续使用中暴露出一组前后端联动问题：

1. 页面刷新后总会回到“首页总览”，无法停留在当前页面。
2. 右上角“任务中心”与“通知”是两个互不感知的弹窗，会出现重叠显示。
3. 从任务中心点击运行中的事件分析任务，会进入空白详情页。
4. 下载的回测报告没有稳定携带运行日志。
5. 事件分析结果页存在参数反复刷新、页面横跳现象。
6. 运行事件分析时出现后端异常：`'numpy.float64' object has no attribute 'dropna'`。

这些问题分布在 `frontend/src/App.jsx`、`frontend/src/api.js`、`backend/services/report_service.py`、`backend/services/event_analysis_service.py`、`event_analysis/result_builder.py` 等关键链路，已经影响页面状态一致性、详情可达性与事件分析可用性。

本方案只做问题梳理与修复规划，不直接进行代码编写。

## 2. 问题清单与初步判断

### 2.1 刷新后跳回首页总览

现象：

- 当前前端刷新后总是回到首页，而不是停留在刷新前所在页面或详情页。

已确认现状：

- `frontend/src/App.jsx` 中 `activeTab`、`selectedBacktestId`、`selectedEventAnalysisId` 均由 React 内存态维护。
- 初始化时 `const [activeTab, setActiveTab] = useState("dashboard");`，没有从 URL、`localStorage` 或其他持久态恢复。
- `refreshAll()` 只刷新数据，不恢复导航上下文。

根因判断：

- 页面路由状态没有持久化。
- 当前是“单页内状态导航”，但没有建立“导航状态 <-> 浏览器地址/持久化存储”的同步机制。

影响：

- 用户刷新、误关页面或浏览器恢复后，工作上下文丢失。
- 详情页、列表筛选页、事件分析页都无法稳定续接。

### 2.2 任务中心与通知弹窗重叠

现象：

- 先点通知，再点任务中心，两个弹窗会同时存在并重叠。

已确认现状：

- `frontend/src/App.jsx` 中 `taskOpen` 与 `noticeOpen` 为两个独立布尔状态。
- 顶部按钮逻辑分别是 `setTaskOpen(!taskOpen)` 与 `setNoticeOpen(!noticeOpen)`。
- 没有互斥关闭逻辑。

根因判断：

- 两个浮层共享同一屏幕位置，但状态机设计成“可同时打开”。
- 缺少全局弹层仲裁机制。

影响：

- 视觉层级混乱。
- 点击与关闭行为不稳定，降低任务状态查看效率。

### 2.3 从任务中心进入运行中事件分析任务时出现空白页

现象：

- 点击任务中心中正在运行的事件分析任务，会跳转到一个空白页面。
- 你的目标是直接进入“事件分析结果页面”，并在任务未完成时也能看到合理内容。

已确认现状：

- `TaskDrawer` 点击事件分析任务时执行 `openEventResult(task.eventAnalysisId)`。
- 实际调用处只做了：
  - `setActiveTab("event_analyses")`
  - `setSelectedEventAnalysisId(id)`
  - `setTaskOpen(false)`
- `EventAnalysisResultView` 在 `analysis.status` 为 `queued/running` 时，不发起结果请求，并把 `payload/task` 清空。
- 该详情页又依赖 `payload.summary`、`payload.details` 等内容，因此运行中任务会进入“空详情壳子”。

根因判断：

- 任务中心跳的是“任务详情视图”，但该视图只按“已完成结果详情”设计。
- 运行中的事件分析缺少独立的“任务过程态详情”。
- 详情页请求条件过于保守，没有利用后端已提供的 runtime 状态与日志兜底返回。

影响：

- 任务中心入口失效。
- 用户无法在运行中查看事件任务进度、参数、日志和最终落点。

### 2.4 下载回测报告时希望附带运行日志

现象：

- 当前下载出的回测报告末尾没有日志内容，希望最终报告携带本次执行日志。

已确认现状：

- 后端 `ReportService.get_report()` 会把 `runtime.logs` 注入接口返回 payload。
- 前端回测详情页也已经能展示 `RuntimeLogPanel`。
- 但 `backend/api/reports.py` 的下载接口只是直接返回物理文件：
  - HTML 直接返回 `report_html_path`
  - JSON 直接返回 `report_json_path`
- 如果生成报告文件时未把日志写进文件，下载内容就不会包含最新日志。

根因判断：

- “页面展示时临时补 runtime.logs” 与 “下载文件内固化 runtime.logs” 是两条不同链路。
- 当前下载依赖静态落盘文件，而不是接口拼装后的增强 payload。

影响：

- 页面和下载件内容不一致。
- 排查问题时，导出的报告不具备完整诊断价值。

### 2.5 事件分析结果页参数反复刷新、页面横跳

现象：

- 打开事件分析结果后，页面参数会反复刷新，表现为来回跳动、重复装载或状态抖动。

已确认现状：

- `App.jsx` 中有一个全局轮询：
  - 每 2.5 秒执行 `Promise.all([refreshBacktests(), refreshEventAnalyses()])`
- `refreshEventAnalyses()` 会整体替换 `eventAnalyses` 数组。
- 详情区使用：
  - `eventAnalyses.find((item) => item.id === selectedEventAnalysisId) || eventAnalyses[0]`
- `EventAnalysisResultView` 的请求依赖是整个 `analysis` 对象：`useEffect(..., [analysis])`

根因判断：

- 轮询刷新时，`analysis` 引用对象频繁变化，即便还是同一个任务 ID，也会触发详情页 effect 反复重跑。
- 当 `find()` 暂时取不到匹配项时，还会退回 `eventAnalyses[0]`，这会造成“详情目标被替换”为另一条记录。
- 详情页的选中态与列表刷新耦合过紧，缺少“按 ID 锁定当前详情对象”的机制。

影响：

- 页面体验上表现为抖动、横跳、参数闪烁。
- 容易造成“看的是 A 任务，结果又切到 B 任务”的错觉。

### 2.6 运行事件分析报错：`numpy.float64` 没有 `dropna`

现象：

- 后端日志显示：

```text
'numpy.float64' object has no attribute 'dropna'
```

- 出错位置在 `event_analysis/result_builder.py`：

```python
series = pd.to_numeric(detail_df.get(col), errors="coerce").dropna()
```

已确认现状：

- `build_summary(detail_df, windows)` 会对每个 `ret_{window}d` 列做汇总。
- 当 `detail_df.get(col)` 不是 `Series` 而是标量值时，`pd.to_numeric(...)` 可能返回 `numpy.float64`，随后调用 `.dropna()` 直接报错。

根因判断：

- 结果汇总层默认假设 `detail_df.get(col)` 恒为 `Series`，但输入结构不稳定时该假设不成立。
- 根本上属于“结果构建阶段对空列/缺列/异常列类型缺少防御性归一化”。

影响：

- 事件分析任务在结果汇总阶段失败。
- 即使扫描过程成功，也无法生成最终统计结果。

## 3. 总体修复目标

本轮修复建议围绕以下五个目标展开：

1. 页面刷新后保留当前导航上下文，包括列表页、详情页与选中记录。
2. 顶部浮层统一互斥，任务中心、通知、账户菜单同一时刻只允许一个打开。
3. 任务中心进入事件分析时，无论任务处于运行中还是已完成，都进入可解释的详情视图。
4. 下载出的回测报告与页面展示保持一致，末尾附带运行日志。
5. 事件分析任务执行链路补强，既修复当前报错，也稳定详情页加载行为。

## 4. 修复方案设计

### 4.1 页面刷新后保留当前页面

目标：

- 刷新后保持在当前模块。
- 如果当前在回测详情或事件分析详情，也应保留对应任务 ID。

推荐方案：

#### 方案 A：基于 URL 查询参数持久化导航状态

建议把以下状态同步到 URL：

- `tab`
- `backtestId`
- `eventAnalysisId`
- 必要时补充部分筛选参数

示例：

- `/?tab=result&backtestId=12`
- `/?tab=event_analyses&eventAnalysisId=8`

优点：

- 刷新天然可恢复。
- 支持复制链接、浏览器前进后退。
- 更符合“页面状态应该可寻址”的 Web 习惯。

需要改造：

- `navigateTo()` 负责同时更新 React 状态与 URL。
- 初始化时从 URL 解析并恢复 `activeTab`、`selectedBacktestId`、`selectedEventAnalysisId`。
- 详情返回列表时，要同步清除对应详情参数。

#### 方案 B：基于 `localStorage` 持久化轻量导航状态

适用条件：

- 如果短期不想引入 URL 级路由管理。

存储内容：

- `activeTab`
- `selectedBacktestId`
- `selectedEventAnalysisId`

优点：

- 改造量较小。

不足：

- 不能分享链接。
- 浏览器前进后退无意义。
- 跨标签页一致性一般。

建议：

- 优先采用方案 A。
- 如果要快速止血，可先落地 B，再在下一轮升级成 URL 路由。

验收口径：

1. 在“回测结果列表”刷新，仍停留在回测结果列表。
2. 在具体“回测详情页”刷新，仍停留在该任务详情页。
3. 在“事件分析详情页”刷新，仍停留在该任务详情页。
4. 无效 ID 时要优雅降级回列表，而不是空白页。

### 4.2 任务中心、通知、账户菜单改成统一互斥浮层

目标：

- 同一时刻只展示一个顶部浮层。

推荐方案：

把以下多个布尔态：

- `taskOpen`
- `noticeOpen`
- `accountOpen`

统一收敛为单个状态，例如：

- `topOverlay = "task" | "notice" | "account" | null`

交互规则：

1. 点击任务中心：
   - 当前不是任务中心时，打开任务中心并关闭其他浮层。
   - 当前已经是任务中心时，关闭。
2. 点击通知：
   - 当前不是通知时，打开通知并关闭其他浮层。
   - 当前已经是通知时，关闭。
3. 点击账户菜单：
   - 同理。
4. `navigateTo()`、点击遮罩、点击关闭按钮时统一清空。

补充建议：

- 增加点击页面空白处关闭浮层。
- 增加 `Esc` 关闭当前浮层。

验收口径：

1. 任务中心与通知不会同时显示。
2. 任务中心与账户菜单不会同时显示。
3. 打开一个再点另一个，旧的自动关闭。
4. 页面切换后浮层全部关闭。

### 4.3 任务中心点击运行中事件分析时的详情页行为重构

目标：

- 从任务中心进入事件分析后，不再出现空白页。
- 运行中的任务也能展示“任务过程态详情”。
- 完成后自动过渡为“结果详情”。

建议拆分为两层视图：

#### 第一层：统一事件分析详情入口

保留 `selectedEventAnalysisId` 作为唯一详情主键。

不论来源是：

- 事件分析列表
- 报告中心
- 任务中心

都统一跳到同一个详情入口。

#### 第二层：详情内部区分状态

按状态拆成三类：

1. `queued/running`
   - 展示任务基本信息
   - 展示参数：日期、窗口、入场规则、去重规则、股票池、过滤条件
   - 展示进度与运行日志
   - 展示“结果尚未生成”的占位
2. `success`
   - 展示完整统计结果、样本明细、运行日志
3. `failed/cancelled`
   - 展示错误信息、运行日志、已知参数
   - 如果没有结果文件，也要能渲染失败详情

后端配合点：

- `ReportService._get_event_analysis_report()` 已经具备“无结果文件时返回 runtime”的雏形。
- 前端不应在 `queued/running` 时直接放弃请求。
- 建议详情页始终尝试读取任务详情/报告接口，由接口返回当前阶段可展示内容。

关键修正点：

1. 取消 `EventAnalysisResultView` 中“运行中就不请求任何数据”的逻辑。
2. 把 effect 依赖从 `[analysis]` 收敛为 `[analysis.id, analysis.status]` 或更稳定的主键依赖。
3. 取消 `find(...) || eventAnalyses[0]` 这种兜底方式，避免跳到错误记录。
4. 如果指定 ID 查不到，应明确显示“任务不存在或已删除”，并提供返回列表入口。

验收口径：

1. 从任务中心点击运行中的事件分析，进入详情页后可见参数、状态和日志，不再空白。
2. 任务完成后，详情页自动显示统计结果。
3. 失败任务能打开并看到错误信息。
4. 删除中的任务或无效 ID 不会导致页面横跳到别的任务。

### 4.4 下载回测报告时附带运行日志

目标：

- 用户下载的报告文件，内容与页面看到的“运行日志”一致。

建议按文件类型拆开处理：

#### JSON 下载

建议把 `runtime.logs` 在任务结束时就正式写入 `report_json_path` 对应文件，而不是只在接口返回时临时补。

好处：

- 下载 JSON 时天然包含日志。
- 页面与下载逻辑共用同一份持久结果。

#### HTML 下载

建议在 HTML 报告模板末尾增加“运行日志”板块，内容来自 `payload.runtime.logs`。

展示建议：

- 默认按时间顺序显示
- 每条展示：
  - 时间
  - 级别
  - 来源
  - 文本内容
- 对超长日志做滚动容器或折叠

接口层建议：

- 如果当前落盘文件未包含最新日志，可在下载前做一次“增强版 payload 重新渲染”。
- 更稳妥的长期方案是：任务成功落盘时就生成“带日志的最终 HTML/JSON”。

注意点：

- 失败任务如果没有 HTML 文件，下载行为应明确限制，只提供 JSON 或仅提供详情查看。
- 成功任务的 HTML 下载按钮应与真实文件状态保持一致。

验收口径：

1. 下载 JSON 后能看到 `runtime.logs`。
2. 下载 HTML 后页面底部存在运行日志模块。
3. 页面展示日志与下载件中的日志条数、内容保持一致。

### 4.5 事件分析结果页反复刷新与横跳修复

目标：

- 列表轮询刷新时，不影响当前详情页稳定性。

建议修复点：

#### 1. 详情锁定按 ID，不按对象引用

当前问题：

- `EventAnalysisResultView` 的 `useEffect` 依赖整个 `analysis` 对象。

建议：

- 只依赖稳定主键，如 `analysis.id`，必要时加 `analysis.status`。
- 避免每次轮询都因为对象引用变化而重拉详情。

#### 2. 移除 `eventAnalyses[0]` 兜底

当前问题：

- 选中项在刷新瞬间拿不到时，会错误回退到第一条任务。

建议：

- 如果 `selectedEventAnalysisId` 对应记录不存在：
  - 优先展示“记录不存在/已删除”
  - 或短暂保留上一份已加载详情
- 不要静默切换到第一条数据。

#### 3. 列表轮询与详情请求解耦

建议：

- 列表轮询只负责更新任务概览。
- 详情页在打开状态下可单独轮询当前任务，周期可略长，例如 3 到 5 秒。
- 详情页只更新当前详情所需的数据，不重新触发整个列表替换带来的联动。

#### 4. 参数区使用稳定快照

如果运行中任务的参数来自列表项对象，应在进入详情页时形成快照，避免轮询时 UI 反复重绘造成视觉抖动。

验收口径：

1. 打开事件分析详情后，页面不再反复闪动。
2. 当前详情不会跳到其他任务。
3. 运行中任务的进度可更新，但参数区不会来回重排。

### 4.6 运行事件分析报错修复方案

目标：

- 解决 `numpy.float64` 的类型错误。
- 提高结果构建层对异常输入的容错性。

根因聚焦：

- `build_summary()` 对 `detail_df.get(col)` 的返回类型没有做强校验。
- 一旦返回标量，就会在 `.dropna()` 处失败。

建议修复策略：

#### 1. 明确列缺失与列存在两种分支

不要直接对 `detail_df.get(col)` 结果链式处理。

建议逻辑：

1. 如果 `col not in detail_df.columns`
   - 直接按空样本窗口输出汇总
2. 如果列存在
   - 强制取 `detail_df[col]`
   - 再做数值化与空值过滤

这样可以避免 `get()` 返回异常类型时进入错误路径。

#### 2. 在汇总前统一列类型

建议对 `ret_{window}d` 列做一次显式归一化：

- 保证列对象是 `Series`
- 非数值统一转 `NaN`
- 再做 `dropna()`

#### 3. 在 `EventAnalysisEngine._attach_returns()` 后增加结果结构校验

建议新增轻量断言：

- 返回表必须是 `DataFrame`
- 所有窗口列都应该存在
- 缺值允许，但列结构必须完整

这样可以把问题前移到更明确的位置，而不是在汇总层才暴露为类型错误。

#### 4. 补测试样例

建议新增至少四组测试：

1. 正常存在收益列
2. 收益列缺失
3. 收益列全部为空
4. 单行样本、部分窗口没有未来价格

验收口径：

1. 重新运行当前事件分析任务，不再出现 `dropna` 异常。
2. 当某个窗口无有效样本时，返回空统计而不是报错。
3. 任务成功时能生成 summary 与 details。

## 5. 推荐实施顺序

建议按风险和收益排序，分三批落地。

### 第一批：先止血

目标：

- 先恢复可用性，减少空白页与任务失败。

范围：

1. 修复 `build_summary()` 的类型处理。
2. 修复事件分析详情页空白问题。
3. 修复顶部浮层互斥。

收益：

- 事件分析可重新运行。
- 任务中心入口恢复可用。
- 顶部交互冲突立即消失。

### 第二批：提升稳定性

范围：

1. 修复事件分析详情横跳与参数反复刷新。
2. 详情页按 ID 锁定，不再兜底跳第一条。
3. 优化运行中任务的过程态详情展示。

收益：

- 详情体验明显稳定。
- 轮询刷新不再干扰用户查看。

### 第三批：补齐产品一致性

范围：

1. 刷新后保留页面状态。
2. 下载报告附带运行日志。
3. HTML 报告追加日志板块。

收益：

- 页面、详情、下载件三者统一。
- 使用体验从“可用”提升到“稳定且可复盘”。

## 7. 风险与注意事项

### 7.1 导航持久化带来的兼容问题

- 如果采用 URL 方案，需要考虑老链接兼容。
- 需要定义无效参数的降级策略，避免用户手动改地址后进入异常状态。

### 7.2 轮询与详情请求叠加

- 当前前端已经存在高频轮询。
- 如果详情页也单独轮询，需要控制频率，避免重复请求过多。

### 7.3 日志写入的一致性

- 页面展示和下载文件都依赖 `runtime.logs`。
- 必须明确“最终以落盘文件为准”还是“接口动态拼装为准”，避免两套逻辑分叉。

### 7.4 运行中任务详情的产品边界

- 运行中的事件分析没有最终 summary/details 是正常的。
- 页面要清楚区分“任务参数/日志”与“分析结果”，避免用户误以为结果丢失。

## 8. 验证清单

### 8.1 导航与刷新

1. 打开回测结果列表，刷新后仍停留原页。
2. 打开某条回测详情，刷新后仍停留该详情。
3. 打开某条事件分析详情，刷新后仍停留该详情。

### 8.2 顶部弹窗互斥

1. 打开通知，再点任务中心，通知自动关闭。
2. 打开任务中心，再点账户菜单，任务中心自动关闭。
3. 页面切换后所有顶部浮层关闭。

### 8.3 事件分析详情

1. 从任务中心点击运行中的事件分析，能看到任务参数与日志。
2. 任务完成后刷新或轮询更新后，能看到统计结果。
3. 失败任务能看到错误信息，不是空白页。

### 8.4 事件分析运行

1. 重跑当前报错任务，不再出现 `numpy.float64` 异常。
2. 多窗口统计可以正常生成。
3. 样本为空时返回空结果，不报错。

### 8.5 报告下载

1. 下载 JSON 报告后包含 `runtime.logs`。
2. 下载 HTML 报告后末尾包含日志区域。
3. 页面看到的日志与下载报告一致。

## 9. 结论

这组问题本质上不是单点 bug，而是三类问题叠加：

1. 前端页面状态缺少持久化与稳定的详情主键设计。
2. 顶部交互缺少统一浮层状态机。
3. 事件分析结果链路在“运行中详情”和“结果汇总容错”上还不够稳。

建议优先修复事件分析报错、顶部浮层互斥和事件任务空白页，再推进详情稳定性与刷新保留，最后补齐下载报告附带日志。这样可以在最短周期内先恢复主要功能可用，再逐步把体验打磨到稳定状态。
